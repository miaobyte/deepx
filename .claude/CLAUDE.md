# 会话边界（最高优先级，违反即终止）

多个模块由不同 Claude 会话并行开发。**严禁跨模块操作。**

## 必须声明

每个会话启动后，**必须先声明以下两项，否则禁止执行任何动作**：

```
会话名: <模块名>
管辖目录:
  - <目录1>    # 可读写，可构建
  - <目录2>    # 可读写，可构建
```

## 边界规则

| 规则 | 说明 |
|------|------|
| **禁止写** | 管辖目录外的一切文件，任何修改、删除、git 操作 |
| **禁止构建** | 管辖目录外的构建命令（`go build`、`make` 等） |
| **可以读** | 管辖目录外的文件（引用接口、理解协议） |
| **禁止 git 全局操作** | `git checkout`、`git reset` 等可能影响其他模块的命令 |

## 联合调试

如需修改管辖目录外的文件（接口对接、联调）：
1. **在会话中明确申请**："申请临时修改 `<路径>`，原因：xxx"
2. 等待用户批准
3. 完成后立即还原或告知

## 违规

一旦发现越界操作，**立即停止**，汇报违规点，不自行修复。

# 术语约定

| 用语 | 行为 |
|------|------|
| 设计方案 / 设计plan | **只设计方案，不写代码** |
| 验证 / 开发 / 实现 | 可以编写代码、构建、测试 |

# 工作流

接到任务时，先在 `doc/plans/<任务名>/` 目录写设计方案，方案确认后再动手写代码。

commit 前检查是否有文件应加入 .gitignore（构建产物、缓存、IDE 临时文件等）。

# 设计哲学

为了追求架构最优，不在意开发成本，可以修改一切。宁为架构破整体，不因妥协留遗憾。

暴露和返回一切错误，拒绝用 try-catch 包庇错误输入。遇错即崩，绝不吞错续行，不让隐性错误潜伏。

# 职责边界原则

每个组件有且仅有一个职责。跨职责的代码即 bug。

## 识别边界

动代码前，先回答三个问题：

1. 这个组件**是什么**？（一句话）
2. 它**只做什么**？（白名单，≤5条）
3. 它**绝不做什么**？（黑名单）

## 违规信号

| 信号 | 说明 |
|------|------|
| 在计算后端里打印结果采样数据 | 计算后端只执行，不验证 |
| 在编排工具里解析业务 DSL | DSL 解析是解释器的职责 |
| 在调度器里管理内存分配 | 内存管理是分配器的职责 |
| 在分配器里执行计算逻辑 | 计算是计算后端的职责 |
| 给组件函数加耗时计时打日志 | profiling 用专业工具，不进业务代码 |
| 格式化输出给"人看" | 展示层（CLI/SDK/前端）的职责 |
| 一个 commit 同时改了多个不相关组件 | 边界模糊，应拆分 |

## 原则

- 宁可跨进程通信，不在一个进程内越界
- 宁可多一次中间层调用，不让组件知道不该知道的事
- 日志只打状态（启动/连接/错误），不打数据（值/采样/耗时）
- 调试信息不应侵入业务代码，用断点、profiler 或外部观察

# 代码风格

极简主义。无冗余变量、无废话注释。名短小精悍。代码即文档。

# 输出约定

始终使用中文交流，包括但不限于：进度汇报、分析过程、方案设计、代码注释、commit message。
文件改动以 tree 格式输出，只列有变动的文件，`#` 后标注改动性质：

```
dir/
├── sub/
│   └── file.py         # 修改|新增|删除: 一句话说明
```
